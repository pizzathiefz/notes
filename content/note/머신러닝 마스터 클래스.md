### 5. Overfitting & Regularization
- 오버피팅의 유일한 판단 기준은 학습 데이터에서 일반 데이터보다 잘하는 것
- 오버피팅의 근본적인 원인은 학습 데이터의 노이즈 = 학습 데이터라는 당장 목격된 현상이 실제 현실을 온전하게 반영하지 못하기 때문에
	- 모델이 복잡할수록 오버피팅되기 쉽다 = 같은 학습 데이터로부터 더 많은 특징을 쥐어짜내는데 그러면 시그널이 아닌 노이즈에 의존할 가능성이 높다
	- 알려진 노이즈라면 인공적으로 제거하거나 단순 학습 데이터를 늘리는 것만으로 (모집단과의 괴리 감소) 개선할 수 있지만, 어떤 방법으로든 노이즈를 완전히 제거할 수는 없기 때문에 결국 모델에 대한 **규제**가 필요
- 오버피팅을 막는 prior 설정
	- MLE는 무조건 목격된 현상(학습 데이터)이 일어날 likelihood가 크도록 파라미터를 찾아냄 -> 그 결과가 오버피팅인 것
	- 모델 likelihood만 최적화하지 말고, 주어진 prior까지 고려해서 posterior를 최적화하라는 미션을 주기
		- 이때 likelihood가 데이터와 모델 파라미터에 따라 결정되는 반면, prior는 데이터와는 상관 없고 모델 파라미터 $\theta$ 만 변수로 가진다.
	- 파라미터 하나하나 개별값이 아닌 전체 파라미터의 특성, 예컨대 파라미터의 norm($\lVert \theta \rVert$)으로 prior를 정의해보자. 
		- $\text{prior}(\theta) = \lambda e ^{-\lambda \lVert \theta \rVert}$ : norm이 작을수록 확률 밀도 값이 높아지는 확률 분포 형태, $\lambda$가 클수록 이 제약 조건을 더 강하게 적용하게 됨 
		- NLL(Negative Log Likelihood) 최적화로 바뀌면 이 prior도 $\lambda \lVert \theta \rVert$로 단순화되며 곱셈이 아닌 덧셈으로 추가됨
	- 이제 모델은 likelihood를 최적화하되 같은 값이면 파라미터를 작게 유지하는 것을 목표로 삼는다.
- 왜 파라미터가 작아지도록 해야 하는가?
	- overdetermined system
		- 맞혀야 할 조건이 변수보다 많을 때 정답이 존재하지 않음 (한 배에 사공이 너무 많아서 모두를 만족시키기 어렵다)
		- 단순한 모델 + 많은 데이터 = 모델이 학습 데이터를 다 설명하지 못함
	- underdetermined system
		- 맞혀야 할 조건보다 변수가 더 많을 때 정답이 무한 개 존재함
		- 복잡한 모델 + 적은 데이터 = 학습 데이터를 설명할 수 있는 여러 모델이 존재할 수 있음
		- 결국 조건을 만족시키는 정답이 무한 개라면 그 중 weight의 norm이 최소화되는 정답을 고르자는 것인데
	- 왜냐면 **파라미터가 클수록 신규 데이터를 만났을 때 크게 틀어짐** (파라미터의 크기는 함수의 변동성과 비례하여 커질수록 바뀌는 데이터에 민감하게 반응함)   
		- ![[assets/머신러닝 마스터 클래스/small-and-big-parameter.png|250]]
- L1과 L2 규제
	- ![[assets/머신러닝 마스터 클래스/l1-l2-norm.png|475]]
	- 둘 다 원점을 향해 끌어당기지만, 궤적이 다름
		- L1은 동일한 크기로 파라미터를 삭감시키다가 더 작은 쪽이 먼저 최소값인 0이 되어버림(먼저 버려짐) = 같은 액수의 세금 = 빈자가 먼저 파산
		- L2는 비중이 큰 파라미터가 더 많이 줄어들긴 하지만 파라미터 간의 비율이 유지되면서 감소 = 같은 비율의 세금
	- 둘 중 무엇을 선택하느냐는 관점에 따라
		- 노이즈에 대한 관점: 주류 위주로 큰 특징들만 남기면 노이즈가 없어질 거야 (L1) vs. 노이즈가 비주류라는 가정을 하기는 어렵고 모든 특징들 다 살리고 크기만 줄이자 (L2)
		- 비용에 대한 관점: (sparsity constraint) 불필요한 특징까지 포함해서 매번 피검자에게 25개의 검사를 다 시킬 거야? 아니면 주요 특징들만 살리는 모델을 만들 거야?
- Batch Normalization과 Weight Decay(L2 Regularization)
	- 배치 정규화란 internal covariate shift(학습 데이터를 배치/미니배치로 나누고 여러 레이어를 통과하면서 분포의 쏠림이 증폭되어 학습을 불안정하게 만듦)문제를 방지하기 위해 매 레이어의 입력값들을 정규화하는 오버피팅 방지 방법
	- 배치 정규화를 하면 weight가 (random walk 특성상 커질 확률이 높지만) 커져도 큰 문제가 없기 때문에 커지는 방향으로 방치된다.
	- 그러나 weight가 커지면 gradient가 감소해서 실질적 학습률(우리가 설정한 학습률에 gradient값을 곱한 것)이 과도하게 떨어지게 되므로, weight decay는 보조적인 역할이 되었지만 여전히 중요하다

### 7. Performance Metrics
- 어떤 상황에서 어떤 지표를 쓰는가
	- 감염 진단 키트에서 sensitivity(= (TP / (TP+FN) = recall), specificity (= TN / (TN + FP) )를 쓰는 이유
		- 똑같은 recall과 spec을 유지하는 상황에서 만약에 감염률이 엄청 올랐다면 precision (= TP / (TP + FP) )은 급상승하게 됨 (즉 같은 키트가 감염률에 따라 성능이 오락가락하면 안 되니 precision은 쓰기 어려움)
	- 그런데 어떤 병이 매우 희귀병이라면? 아니면 키워드에 맞는 문서 대비 관련 없는 문서가 무한히 많은 정보 검색 분야라면?  -> recall, precision
		-  매우 imbalanced한 dataset = 낮은 양성(감염률)이 유지되는 상황
		- TN이 매우 거대할수록 spec을 지배하게 되어 spec의 값은 거의 차이나지 않게 높은 값을 유지하지만 precision은 TP와 FP에 따라서(실제로 정말 분류를 잘했느냐) 오락가락할 수 있음
- 어떤 상황에서 어떻게 임계치를 정하는가
	- ![[assets/머신러닝 마스터 클래스/threshold-and-metrics.png|325]]
	- sense(recall)과 spec은 임계치에 따라 각각 역방향, 순방향으로 증가하고 precision은 N,P 분포 둘 다를 고려하기 때문에 분포 겹치는 정도에 따라 달라질 수 있음
	- 임계치를 어떻게 잡느냐 또한 상황에 따라 달라짐
		- 치사율이 높지만 조기 발견하면 완치 가능하고 2차 검사 비용이 싼 질병 vs. 걸렸을 때 큰 문제는 없지만 감염루트가 남부끄럽고 2차 검사 비용이 비싼 질병
			- 전자는 recall이 중요하고 후자는 spec이 중요
		- 검색의 경우 사용자는 recall을 집작할 수 없지만 검색어와 관련 없는 결과가 나오는 건 바로 체감 = precision을 높이는 게 중요
- AUC
	- 분류 모델의 성능을 나타낼 때는 주로 하나에 임계치에 따른 지표 결과보다는 여러 임계치에 대한 성능을 뽑고 이를 종합하는 방식을 사용함 
	- ROC curve = 임계치에 따라 달라지는 (FPR, TPR) 쌍을 점으로 표현
	- PR curve = 임계치에 따라 달라지는 (recall, precision) 쌍을 점으로 표현
		- ROC curve와 달리 N,P 분포도 상황에 따라 모양이 다양할 수 있음
	- ![[assets/머신러닝 마스터 클래스/roc-pr-curve.png|350]]
	- 이런 커브 아래의 면적 넓이를 AUC라고 하고 이 값이 클수록 좋은 성능
	- 심한 클래스 불균형 상황에서 ROC는 변별력이 없음 (negative가 너무 거대하기 때문에)
- AP
	- 검출 모델은 분류 모델과 조금 다르다
		- 임계값에 따라 음성/양성 구분이 아닌 모델이 나름대로 계산한 순서가 있는 정답 후보 N개를 뱉고 그 안에 실제로 정답이 들어 있는지에 따라 평가
